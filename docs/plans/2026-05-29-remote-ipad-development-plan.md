# Remote iPad Development Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Make SeleneMarkup fully developable from a MacBook Air + iPad while traveling, with the home Mac mini serving Selene over Tailscale against an isolated dev database.

**Architecture:** The MacBook Air (Xcode) + iPad are a self-contained build/deploy station via `./redeploy.sh --dev` (USB). The Mac mini stays home running a persistent `SELENE_ENV=development` server on `:5679` against `~/selene-data-dev/selene.db` (seeded from a prod snapshot), reachable from anywhere over Tailscale (`100.111.6.10`). The only code change is the baked-in IP in `AppConfig.swift`'s `#if SELENE_DEV` branch. The prod server (`~/selene-prod`, `:5678`) is never touched.

**Tech Stack:** Swift/SwiftUI + xcodegen + xcodebuild/devicectl (SeleneMarkup); TypeScript/Fastify + ts-node (selene server); launchd; Tailscale; SQLite; git/GitHub.

**Two repos, two locations:**
- `~/selene` — server + launchd agent (this repo)
- `~/SeleneMarkup` — the iPad app (separate repo, no remote yet)

**ORDERING CONSTRAINT:** Phases 1–5 MUST be done on the Mac mini *with physical access to the iPad* (USB) before leaving. Phase 6 is the away-from-home verification.

**Reference design:** `docs/plans/2026-05-29-remote-ipad-development-design.md`

## Phase 3: Persistent dev-mode server (launchd agent)

Goal: a `SELENE_ENV=development` server on `:5679` that survives reboot/sleep. Model it on the existing prod server plist.

### Task 3.1: Read the installed prod server plist as the template

**Files:** Read `~/Library/LaunchAgents/com.selene.prod.server.plist` (the live, working agent).

**Step 1:**
```bash
cat ~/Library/LaunchAgents/com.selene.prod.server.plist
```
Expected (verified 2026-05-29): runs `/usr/local/bin/node /Users/chaseeasterling/selene-prod/dist/server.js`
with an `EnvironmentVariables` dict setting `PATH`, `SELENE_DB_PATH`, `SELENE_ENV=production`,
plus `KeepAlive`, `RunAtLoad`, `WorkingDirectory`. **It uses an absolute node path and a
compiled `dist`, NOT `npx ts-node`** — launchd runs with a minimal environment, so the dev
agent must also use absolute paths and an explicit env.

**Step 2:** Confirm the node binary path exists (prod uses `/usr/local/bin/node`):
```bash
ls -l /usr/local/bin/node && /usr/local/bin/node -v
```
Expected: a node binary + version. If it's missing, use `which node` and substitute that
absolute path everywhere below.

**Step 3:** Confirm the ts-node bin exists in the dev repo (the dev server runs from source):
```bash
ls -l ~/selene/node_modules/.bin/ts-node
```
Expected: the ts-node launcher script exists. (It's a node script, so `node <path>` runs it.)

### Task 3.2: Author the dev server plist (mirror prod's absolute-path style)

**Files:**
- Create: `~/selene/launchd/com.selene.dev.server.plist`

**Step 1:** Create the plist. Running source under launchd = `node <ts-node bin> src/server.ts`
with everything absolute and the env set explicitly (do NOT rely on `npx` or shell PATH):
```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>EnvironmentVariables</key>
  <dict>
    <key>PATH</key>
    <string>/usr/local/bin:/opt/homebrew/bin:/usr/bin:/bin:/usr/sbin:/sbin</string>
    <key>SELENE_ENV</key>
    <string>development</string>
    <key>SELENE_DB_PATH</key>
    <string>/Users/chaseeasterling/selene-data-dev/selene.db</string>
    <key>PORT</key>
    <string>5679</string>
  </dict>
  <key>Label</key>
  <string>com.selene.dev.server</string>
  <key>ProgramArguments</key>
  <array>
    <string>/usr/local/bin/node</string>
    <string>/Users/chaseeasterling/selene/node_modules/.bin/ts-node</string>
    <string>/Users/chaseeasterling/selene/src/server.ts</string>
  </array>
  <key>WorkingDirectory</key>
  <string>/Users/chaseeasterling/selene</string>
  <key>RunAtLoad</key>
  <true/>
  <key>KeepAlive</key>
  <true/>
  <key>StandardOutPath</key>
  <string>/Users/chaseeasterling/selene/logs/dev-server.out.log</string>
  <key>StandardErrorPath</key>
  <string>/Users/chaseeasterling/selene/logs/dev-server.err.log</string>
</dict>
</plist>
```
NOTE: `SELENE_DB_PATH` is set explicitly to the dev DB (belt-and-suspenders; `config.ts` also
derives it from `SELENE_ENV=development`). The `.env`/`.env.development` files are also loaded by
`config.ts` at startup regardless of launchd env. If `node_modules/.bin/ts-node` doesn't run
cleanly under launchd, the fallback is to compile (`npm run build` if defined) and point
`ProgramArguments` at `node dist/server.js` exactly like prod.

**Step 2:** Validate the plist syntax:
```bash
plutil -lint ~/selene/launchd/com.selene.dev.server.plist
```
Expected: `OK`.

### Task 3.3: Install and start the agent

**Step 1:** Symlink/copy into LaunchAgents and load (match how `scripts/install-launchd.sh` does it — prefer reusing that script if it auto-discovers plists):
```bash
cp ~/selene/launchd/com.selene.dev.server.plist ~/Library/LaunchAgents/
launchctl bootstrap gui/$(id -u) ~/Library/LaunchAgents/com.selene.dev.server.plist 2>/dev/null \
  || launchctl load ~/Library/LaunchAgents/com.selene.dev.server.plist
```
Expected: no error.

**Step 2:** Confirm it's running:
```bash
launchctl list | grep com.selene.dev.server
```
Expected: a line with a PID (not just `-`).

**Step 3:** Health check against the dev server:
```bash
curl -s http://localhost:5679/health
```
Expected: a healthy JSON response with `env: development`, `port: 5679`.

**Step 4:** Confirm it serves real cluster data (the app's entry point). The cluster-list
endpoint is `GET /api/clusters` (verified in `src/routes/notes.ts`). Auth is **bypassed** on
the dev server because `SELENE_API_TOKEN` is unset (`requireAuth` returns early when
`config.apiToken` is empty), so no bearer header is needed:
```bash
curl -s http://localhost:5679/api/clusters | head -c 400
```
Expected: `{"clusters":[...]}` — non-empty (83 clusters from the prod snapshot). If you later
set `SELENE_API_TOKEN` in dev, add `-H "Authorization: Bearer <token>"` here AND set the same
token in the app via the `selene_bearer_token` UserDefaults key.

### Task 3.4: Verify survival across restart

**Step 1:** Kick the agent to simulate a restart:
```bash
launchctl kickstart -k gui/$(id -u)/com.selene.dev.server
sleep 3 && curl -s http://localhost:5679/health
```
Expected: server comes back up and responds. (A full reboot test is ideal before leaving if time permits.)

### Task 4.2: Disable sleep persistently

**Step 1:** Prevent system + disk sleep and enable wake-for-network (Mac mini is on AC, so use the system settings, not a session-bound `caffeinate`):

**Step 2:** Also set in System Settings → Energy/Battery: "Prevent automatic sleeping when display is off" / "Wake for network access" as a belt-and-suspenders.


Expected: `Remote Login: On`.

## Notes for the executor

- **Free Apple ID signing expires after ~7 days** — if the Dev app stops launching mid-trip, just re-run `./redeploy.sh --dev` from the laptop (iPad on USB). A paid Apple Developer account avoids this.
- **If `npx ts-node` fails under launchd** (minimal PATH), switch the plist to an absolute node + a compiled entry, or a wrapper script — mirror exactly what the prod server plist does.
- **Don't run `create-dev-db.sh` during the trip** — it would rebuild the dev DB without the prod snapshot (and could drop `source_note_id`).
- **MagicDNS alternative:** `chases-mac-mini.taila69703.ts.net` works in place of `100.111.6.10` if MagicDNS is enabled on all devices; the raw IP is more robust for a baked-in URL.
